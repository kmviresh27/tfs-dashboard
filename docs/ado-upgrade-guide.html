Overview
Good news: The AV Dashboard is well-architected for this migration. Every API call reads baseUrl, apiVersion, and pat from config.json at runtime — nothing is hardcoded. The migration is essentially a config-only change.
PAT required: Personal Access Tokens do not migrate from TFS to ADO Server 2021. You must generate a new PAT in the new server and update config.json before the dashboard will connect.
Impact Summary
Area Impact Action Required
Server URL (tfs.baseUrl) Medium Update if server hostname changes. On-prem URL pattern is identical: https://<server>/tfs/<Collection>/<Project>
Personal Access Token (tfs.pat) Required Generate new PAT in ADO Server 2021 — old PATs do not carry over
API Version (tfs.apiVersion) Low Update from 5.0 → 6.0. Both work; 6.0 unlocks new response fields
Custom Fields Verify Confirm Custom.Defects.*, Custom.Planning.Release, etc. migrated with same names
Work Item Types Low Verify Feature, Defect, Objective, Story types still exist with same names in process template

WIQL Query Syntax None Fully backward compatible — no changes needed
REST API endpoints (_apis/wit/*) None All endpoints used by dashboard exist in ADO Server 2021
PAT Authentication (Basic Auth) None Same mechanism — Authorization: Basic base64(:PAT)
Work Item deep links (/_workitems/edit/{id}) None URL pattern unchanged in ADO Server 2021
Iteration / Area path structure None Classification nodes API (_apis/wit/classificationnodes) is unchanged
Snapshots folder None Local JSON files — completely unaffected by server migration
Team capacity endpoints None _apis/work/teamsettings/iterations and capacities — unchanged
Config Changes Required

Open config.json in the dashboard root directory and update the following three values:

KeyBeforeAfterNotes
tfs.baseUrl https://tfs.yourorg.com/tfs/YourOrg/YourProject <new ADO Server URL> Same format: https://<host>/tfs/<Collection>/<Project>
tfs.pat <old PAT> <new PAT from ADO 2021> Generate in ADO Server 2021 → User Settings → Personal Access Tokens
tfs.apiVersion 5.0 6.0 Optional but recommended — enables newer response features
Tip: After updating config.json, use the dashboard's built-in Settings page → TFS Connection tab and click Test Connection to validate before opening the full dashboard.
Items to Verify After Migration
🔬 Custom Fields

These custom org fields must exist with identical names in the migrated process template. Open any Feature or Defect in ADO Server 2021 and confirm these fields are present:

Field NameUsed InDashboard Section
Custom.Planning.ReleaseFeatures, StoriesFeatures, Sprint, PI Checks
Custom.Defects.ClassificationDefectsQuality — filters out Enhancements
Custom.Defects.ProjectDefectsQuality — Field Defects by Project
Custom.Defects.FixedPlannedVersionDefectsQuality — Versions tab
Custom.Defects.WhereFoundDefectsQuality — Analysis tab
Custom.Generic04DefectsResolve-by date (SLA breach)
Custom.RankDefectsSeverity / ranking charts
If a field is missing: The dashboard's fetchWorkItemDetails function has automatic field-stripping fallback — it will log a warning and retry without the unknown field. The section will still load, but that column will be blank.
📂 Saved Query Paths (PI Checks)

The PI Checks section fetches queries from a fixed folder path in TFS Shared Queries. After migration, verify this path still exists:

Shared Queries / ICAP / Program Queries / TFSInconsistenciesQueryRepository

If the path changed, update it in Settings → PI Checks → Query Folder Path.

🧪 Test Coverage Section
The Test Coverage section calls /_apis/test/runs. Test run history may not be fully migrated. Load the Test Coverage tab and verify data appears. If empty, test runs may need to be re-executed against the new server.
🔤 Work Item Type Names

These work item type names are configured in Settings → Field Mappings. Verify the process template in ADO 2021 uses the same names:

Feature · Story · Defect · Objective · Risk · Product Risk · Test Case · Epic · Task

If any were renamed during migration, update them in the Settings page.

Migration Checklist
Work through these steps in order on migration day. Estimated time: ~1 hour.
  • Get the new ADO Server 2021 URL from IT/admin
    Format: https://<new-hostname>/tfs/YourOrg/Healthcare%20IT
  • Generate a new Personal Access Token in ADO Server 2021
    User Settings (top right) → Personal Access Tokens → New Token → Scopes: Work Items (read), Code (read), Test (read)
  • Update config.json — set tfs.baseUrl, tfs.pat, tfs.apiVersion
    See the Config Snippet section below for exact format
  • Restart the AV Dashboard server
    node server.js from the dashboard root directory
  • Open Settings → TFS Connection → click Test Connection
    Should show green ✅. If red, double-check URL format and PAT scopes.
  • Navigate to Overview — confirm PI list loads and data appears
    Verifies WIQL + workitemsbatch + classificationnodes all work
  • Check Quality → Versions tab — confirm defect data loads
    Verifies Custom.Defects.FixedPlannedVersion and other custom fields exist
  • Check PI Checks — confirm queries load from Shared Queries folder
    Path: Shared Queries / ICAP / Program Queries / TFSInconsistenciesQueryRepository
  • Check Test Coverage tab — confirm test run data appears
    If empty, test runs may need re-execution on the new server
  • Check Sprint Capacity — confirm team iteration capacity loads
    Uses _apis/work/teamsettings/iterations/{id}/capacities
  • Verify TFS deep links open correctly
    Click any work item ID in the dashboard — should open in ADO Server 2021
  • Take a new baseline snapshot for Scope Change Tracking
    Old snapshots remain valid but reference the old server URL in TFS links
Updated config.json — TFS Section

Only the three highlighted lines need to change. All other settings remain the same.

// config.json — tfs section (update highlighted values only) "tfs": { "baseUrl": "https://<new-ado-server>/tfs/YourOrg/Healthcare%20IT", // ← UPDATE "pat": "<new-PAT-from-ADO-2021>", // ← UPDATE "apiVersion": "6.0", // ← UPDATE (was 5.0) "organization": "YourOrg", // unchanged "project": "Healthcare IT", // unchanged "areaPath": "Healthcare IT", // unchanged "iterationPath": "Healthcare IT\\ISP" // unchanged }
Troubleshooting
SymptomLikely CauseFix
Dashboard shows "PAT not configured" or 401 Old PAT used, or PAT not yet saved Generate new PAT in ADO 2021 → update tfs.pat in config.json → restart server
All sections show "No data" but no error Wrong baseUrl — connection succeeds but returns empty results Double-check Collection and Project name in the URL. Try opening the URL in a browser.
PI Checks tab shows query folder not found Query folder path changed during migration Locate the query folder in ADO 2021 → update path in Settings → PI Checks
Quality / Versions tab shows no defect fields Custom.Defects.* fields missing from process template Check server logs for "Stripping unknown field" warnings. Contact ADO admin to verify custom field migration.
Test Coverage section is empty Test runs not migrated or not yet executed Run test suites in ADO 2021 to generate test run history. Historical runs may not migrate.
TFS work item links go to old server Old baseUrl still saved in config.json Ensure tfs.baseUrl is updated and server is restarted
Snapshot Scope Change shows stale data Snapshots reference old server — compare still works but TFS links point to old URL Take new snapshots after migration for accurate TFS links in scope change tracking
Server logs: Run node server.js from a terminal and watch the console output. All TFS API errors are logged with the full URL and HTTP status code to help diagnose connection issues quickly.
