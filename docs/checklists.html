🆕 Fresh Installation
For first-time setup on a new machine
0 / 0
🖥Prerequisites
Run node --version — must show v18 or higher. Download from nodejs.org.
Test with: curl https://dev.azure.com/YourOrg (or your TFS URL). Should return HTML/JSON, not a timeout.
ADO: User Settings → Personal Access Tokens. Required scopes: Work Items (Read). Copy and store the token securely — it won't be shown again.
In ADO: Project Settings → Boards → Area / Iterations. Note the exact paths including backslash separators (e.g. MyProject\TeamA).
📦Installation
E.g. C:\apps\av-dashboard\. Avoid paths with spaces or special characters.
Wizard creates config.json, initialises data/, and creates the super-admin account.
Verify baseUrl, pat, areaPath, iterationPath. Change sessionSecret if setup didn't auto-generate one.
Watch the console for "Server started on port 3000" and no connection errors on first data fetch.
Use the super-admin email and password set during setup.
🔍Verification
Navigate to Overview or Features. If charts show zeros, go to Settings → Connection and click Test Connection.
Check the PI pills in the top bar. If PIs are missing, verify piNamingPattern in Settings → PI Config matches your ADO iteration names.
Settings → Branding: set company name, app name, logo, and primary colour to match your organisation.
Run .\scripts\deploy.ps1 to install as a Windows service via NSSM. Service name: AVDashboard.
Run as admin: New-NetFirewallRule -DisplayName "AV Dashboard" -Direction Inbound -Protocol TCP -LocalPort 3000 -Action Allow
✅ All items complete!
🔄 TFS → Azure DevOps Migration
For teams moving from on-prem TFS to Azure DevOps cloud (or vice versa)
0 / 0
📋Pre-Migration Assessment
Format: https://dev.azure.com/{org}/{project}. Confirm the project name matches exactly (case-sensitive).
Old TFS PATs do not work with Azure DevOps cloud. Generate a new PAT at dev.azure.com with Work Items (Read) scope.
In ADO: Project Settings → Boards → Areas. The area path structure may differ from your TFS instance.
Check ADO: Project Settings → Boards → Iterations. Sprint naming may have changed (e.g. from "PI1 S1" to "PI1-SP1"). Note the exact format.
ADO cloud process templates may use different type names (e.g. "User Story" instead of "Feature"). Check in ADO Project Settings → Process.
If your TFS used custom fields (e.g. for defect escape, severity), verify they exist in ADO under the new process template with the same field reference names.
⚙Configuration Update
In Admin → Department → Connection (or data/departments/{id}/config.json): change baseUrl to the new ADO URL.
Azure DevOps cloud uses API version 6.0. Change in the department config or Settings → Connection.
Admin → Department → Connection → PAT field. Or edit data/departments/{id}/config.json directly (restart server after).
The ADO area path root may differ from TFS. Verify it with a test WIQL query in ADO.
ADO projects often use different PI naming. Update piNamingPattern (e.g. PI{yy}.{n}) and sprintSubpathPattern (e.g. {pi}-{sprint}) to match.
If sprint labels changed (e.g. from S1/S2/S3 to SP1/SP2/SP3), update sprintLabels in Settings → PI Config or department config.
Settings → Field Mappings. If ADO uses "User Story" instead of "Feature" or "Bug" instead of custom defect type, update these.
Some ADO 2021 process templates use a custom "Planned For" field instead of System.IterationPath. If features show as "Unassigned", set this field in Settings → Field Mappings.
✅Post-Migration Verification
Click "Test Connection" — should return list of teams/area paths from the new ADO project.
Navigate to Features and Defects sections. Charts should show non-zero data for recent PIs.
Check Velocity section — sprint data should match what you see directly in ADO for recent sprints.
If older PIs don't exist in the new ADO instance, they will return empty data (not errors) — this is expected. Check the browser console for unexpected 500 errors.
Once migrated, revoke the old TFS PAT from your TFS instance to reduce attack surface.
Snapshots captured against the old TFS instance are still stored under data/departments/{id}/snapshots/. They may not match ADO data. Re-capture if needed.
✅ Migration checklist complete!
🏢 New Department Onboarding
Adding a new department (tenant) to an existing AV Dashboard deployment
0 / 0
📋Information Gathering
E.g. eng-platform, qa-team, mobile-app. Lowercase, hyphens OK. Cannot be changed later without manual file renaming.
Need: base URL, PAT (Work Items Read), area path, iteration path, API version.
Check how PIs appear in the ADO iteration tree. May differ from other departments.
Who will manage this department's users and settings? They need an account and the "admin" role for this department.
⚙Configuration
Fill in all connection fields. Click Save — this creates data/departments/{id}/config.json.
Admin → Department → Connection → Test Connection. Must return team list without errors.
Settings → PI Config for this department. Verify PIs appear in the top bar after saving.
Settings → Field Mappings. Update featureType, defectType, and custom fields as needed.
Settings → Branding. Set the department display name and optionally a logo or primary colour.
Settings → RAG Thresholds. Different teams may have different quality expectations.
👥Users & Access
Admin → Department → Members → Add member → set role to "admin".
Settings → Members (as dept admin) or Admin → Department → Members. Assign appropriate roles.
Settings → Policies. Hide sections that this team doesn't use (e.g. Test Coverage, Objectives).
URL format: http://server:port/d/{dept-id}/. Share login credentials or Azure AD SSO instructions.
✅ Department onboarding complete!
🚀 Production Go-Live
Before launching AV Dashboard to end users in production
0 / 0
🔒Security
Settings → Profile or Admin → All Users. Use a password manager to generate a strong password (min 16 chars).
Check config.json → app.sessionSecret. Must be a long random string (32+ chars). Setup wizard auto-generates one.
config.json and data/ must be in .gitignore. Never commit them to version control.
Do not expose port 3000 to the internet. Use IIS/nginx as a reverse proxy with HTTPS, or restrict access to corporate VPN.
Review Admin → All Users. Remove test accounts. Ensure no accounts share passwords.
🏗Reliability
Run .\scripts\deploy.ps1. Verify service is running: Get-Service AVDashboard. Test by rebooting and checking the dashboard comes back up.
Schedule regular backups of the data/ folder (contains user accounts, dept configs, PI snapshots). A daily xcopy or robocopy to a shared drive is sufficient.
NSSM writes logs to server-out.log and server-err.log. Set NSSM log rotation or schedule a cleanup task to prevent disk fill.
Azure DevOps PATs expire. Set a calendar reminder 2 weeks before expiry to regenerate and update config.json.
Lower values = more ADO queries. For large programmes with 200+ features, use 30 minutes or higher.
🎯Readiness
Switch through each department via the top bar department switcher. Features and defects should load for each.
Settings → Notifications. Add the Teams incoming webhook URL to receive daily digests and anomaly alerts.
Share http://server:port/docs/user-manual.html or the ? help button with all users. Brief users on the PI filter and team filter in the top bar.
Decide who is responsible for taking the PI planning snapshot (📸 Snap button) at PI planning completion. This enables predictability tracking.
✅ Ready for go-live!
⬆️ Version Upgrade
Upgrading an existing AV Dashboard installation to a new version
0 / 0
💾Pre-Upgrade
Copy data/ to a safe location. This contains all user accounts, dept configs, and PI snapshots.
Copy config.json to config.json.bak. New version may add config fields.
Check the CHANGELOG or release notes for any config schema changes, renamed fields, or required migration steps.
Notify users before taking the service down for upgrade. Most upgrades take 2–5 minutes.
⬆Upgrade Steps
Or: .\scripts\undeploy.ps1 if using NSSM. Wait for the process to fully exit.
Extract the new ZIP, overwriting src/, server.js, client/dist/, and scripts/. Do NOT overwrite data/, config.json.
Updates production dependencies. Run from the deployment folder, not the source folder.
Compare your config.json with config.sample.json from the new version. Add any missing fields with their default values.
Or: .\scripts\deploy.ps1 -SkipBuild to re-register service. Check event log for startup errors.
✅Post-Upgrade
Verify you can log in with existing accounts. User data in data/users.json should be intact.
Switch through each department and verify Features/Defects sections load without errors.
Click 📸 Snap → browse existing snapshots. They should still be listed and loadable.
Once everything is confirmed working, delete the backup copies of data/ and config.json.bak.
✅ Upgrade complete!
👤 User Onboarding
For department admins welcoming new users to AV Dashboard
0 / 0
🔑Account Setup
Set email, display name, temporary password, and role (member / admin).
Use a secure channel (e.g. encrypted email, Teams private message). Ask user to change password on first login.
Include the direct department URL if multi-tenant: http://server:port/d/{dept-id}/
🎓Orientation
These two filters are the most-used controls. Show how to select PIs and filter to a specific team.
Clicking chart bars/pie slices opens the corresponding filtered work item query in Azure DevOps. Very useful for drill-down.
The ? button in the top bar opens the in-app help panel with search. Full user manual at /docs/user-manual.html.
📸 Snap button in top bar captures the current PI state. Predictability section uses this for tracking.
Ask the user to confirm they can see the sections they need. If something is hidden, check Settings → Policies.
✅ User onboarding complete!
