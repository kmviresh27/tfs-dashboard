🚀 New System Deployment Checklist

Complete every REQUIRED item before going live. Optional items improve experience but are not blocking.

1️⃣
Infrastructure & Server
Before launching the Node.js process
  • Node.js 18+ installed — verify with node -vREQUIRED
  • Run npm install in the project root to install all dependenciesREQUIRED
  • Run npm run build to compile the React client into dist/REQUIRED
  • Confirm port 3000 is free or set PORT env variableREQUIRED
  • Start server — node server.js (or via PM2 / Windows Service)REQUIRED
  • Verify config.json is writable in the project root (server writes settings here)REQUIRED
  • Set up process manager (PM2, NSSM, Windows Task Scheduler) for auto-restartOPTIONAL
2️⃣
TFS Connection
Settings → TFS Connection tab
  • Enter TFS Base URL — e.g. https://tfs.company.com/tfs/CollectionREQUIRED
  • Generate a PAT with Work Items (Read) scope in TFS and paste itREQUIRED
  • Set Area Path — root area path of the programme (e.g. Healthcare IT\ICAP\ISP)REQUIRED
  • Set Team Root Path(s) — one line per team parent path. Each line must be one level above team nodesREQUIRED
  • Set Iteration Path — root iteration used for PI/Sprint queriesREQUIRED
  • Click "Test Connection" — confirm team list is returned with ✅REQUIRED
  • Save TFS SettingsREQUIRED
3️⃣
Field Mappings
Settings → Field Mappings tab — adapt to your process template
  • Work Item Types — confirm or adjust names for Feature, Story, Defect, Objective, Risk, Epic, TaskREQUIRED
  • Effort / Story Points fields — set TFS reference names for effort and story points fieldsREQUIRED
  • Release field — set the custom release grouping field reference nameREQUIRED
  • Defect Classification field — for enhancement exclusion logicOPTIONAL
  • Feature Done / Removed states — single-value state stringsREQUIRED
  • Feature WIP + All States — comma-separated listsREQUIRED
  • Story Done + WIP + Removed — comma-separatedREQUIRED
  • Defect Closed + Removed statesREQUIRED
  • Sprint Labels — e.g. S1, S2, S3, IPREQUIRED
  • PIs Per Year — default 4REQUIRED
  • PI Naming Pattern — e.g. {yy}-PI{n} → 26-PI1REQUIRED
  • Hours per Man-Day — default 6. Adjust if your team works different hoursOPTIONAL
  • Save Field MappingsREQUIRED
4️⃣
RAG Thresholds
Settings → RAG Thresholds tab
  • Review Done Rate thresholds — default Green ≥ 80%, Amber ≥ 50%OPTIONAL
  • Review Escape Ratio — default Green ≤ 20%, Amber ≤ 40%OPTIONAL
  • Adjust thresholds to match your programme's targets, then SaveOPTIONAL
5️⃣
Advanced & Branding
Settings → Advanced + Branding tabs
  • Programme Start Year — used for PI year calculations (Advanced tab)REQUIRED
  • Data Refresh Interval — default 30 min. Set lower for high-frequency data, higher to reduce TFS loadOPTIONAL
  • Company / App Name — Branding tab, shown in title bar and sidebarOPTIONAL
  • Primary Color — colour picker for accent. Default accent blue #1492ffOPTIONAL
  • Logo — URL or inline SVG. Clear both fields to use text brandingOPTIONAL
6️⃣
Authentication (Admin-only)
Settings → Azure AD + Role Mappings (admin role required)
  • Register app in Azure AD — note Tenant ID, Client ID, generate Client SecretADMIN
  • Set Redirect URL — must match the callback URL registered in Azure AD appADMIN
  • Save Azure AD settings and restart the server to apply OIDC clientADMIN
  • Add Role Mappings — map Azure AD groups/email patterns to dashboard roles (exec, rte, pm, sm, admin)ADMIN
  • Configure Admin Roles — roles with full settings accessADMIN
7️⃣
Notifications
Settings → Notifications tab
  • Paste MS Teams / Slack webhook URL for digest deliveryOPTIONAL
  • Set Digest Schedule — day + hour of weekly digestOPTIONAL
  • Choose Digest Sections — Delivery, Quality, Forecast, Risks, VelocityOPTIONAL
  • Test Webhook — click "Send Test" to confirm deliveryOPTIONAL
  • Set Threshold Alerts — configure metric + role + threshold for real-time alertsOPTIONAL
  • Save & Enable toggle notifications onOPTIONAL
8️⃣
Visibility Policies (Admin-only)
Settings → Visibility tab — control what each role sees
  • Select each role (All, Exec, RTE, PM, SM) and configure visible sections, including Scope Change where appropriateADMIN
  • Expand section cards to toggle individual tabs and chart cards per roleADMIN
  • Save Policies — changes take effect immediately on next page loadADMIN
✅ Deployment complete when: TFS Test Connection passes, data loads on dashboard, at least one PI is visible in PI Delivery.
👥 New Team Onboarding Checklist

Use this when adding a new Agile Release Train team to an already-running system.

🔌
TFS Paths
Settings → TFS Connection
  • Add new Team Root Path — open TFS Connection, add one new line for the team's parent pathREQUIRED
  • Confirm Area Path includes the new team's work itemsREQUIRED
  • Confirm Iteration Path includes the new team's sprint iterationsREQUIRED
  • Test Connection — new team should appear in the returned team listREQUIRED
  • Save & reload dashboard — new team should appear in Team selector dropdownsREQUIRED
🧭
Field Mapping Validation
Settings → Field Mappings — verify the new team uses same process template
  • Check Work Item Type names — confirm new team's TFS uses the same type names (Feature, Story, etc.)REQUIRED
  • Check custom field reference names (Effort, Release, Severity, etc.) — if team uses different names, update mappingsREQUIRED
  • Check State Values — new team's Done/WIP/Removed state names must match configured valuesREQUIRED
  • Check Sprint labels — confirm PI sprint labels match iteration naming for the new teamREQUIRED
👤
User Access
Settings → Role Mappings (Admin)
  • Add role mappings for the new team's Scrum Master → sm roleADMIN
  • Add role mappings for the new team's PM / PO → pm roleADMIN
  • Verify Visibility Policies — confirm the relevant roles can see required sectionsADMIN
✅
Validation
Verify data is visible for the new team
  • PI Delivery page — new team's features should appear in current PIREQUIRED
  • Velocity page — team selector includes new teamREQUIRED
  • Sprint page — sprints load for the new team's iterationsREQUIRED
  • Team Summary (Features page) — new team row appears in capacity chartREQUIRED
  • Defects page — defects for the new team are visibleOPTIONAL
🔌 TFS Connection

Configures how the dashboard authenticates to and queries Azure DevOps Server (TFS). All dashboard data originates from these settings.

FieldDescriptionDefaultExample
TFS Base URL Root URL of the TFS collection. Must be reachable from the server process. — https://tfs.company.com/tfs/DefaultCollection
Personal Access Token PAT with Work Items (Read) scope. Stored encrypted in config.json. Leave blank on re-save to keep existing value. — abc123xyz…
Area Path Root area path that scopes all WIQL queries. Must contain all teams' work items. — Healthcare IT\ICAP\ISP
Team Root Path(s) One path per line. Each path is one level above individual team nodes. Used to discover team list. — Healthcare IT\ICAP\ISP\Hercules
Iteration Path Root iteration path for PI/Sprint lookups. All sprint iterations must live under this path. — Healthcare IT\ICAP\ISP
⚠️ PAT Rotation: When a PAT expires, open TFS Connection, paste the new PAT, and save. The old value is automatically replaced. All PATs are stored server-side only.

Test Connection Button

Clicking 🔌 Test Connection calls /api/teams and displays the returned team list. A successful test confirms: Base URL reachable, PAT valid, Area Path correct. Always test after any TFS change.

Threshold Alerts

Real-time alerts fired when a metric crosses a configured threshold. Each alert can target a different webhook URL.

FieldDescription
Alert Webhook URLSeparate webhook for immediate alerts. Can be different from the digest URL.
Anomaly ThresholdMultiplier applied to standard deviation for anomaly detection. Default 1.5×.
Threshold RulesAdd one rule per metric — choose metric, operator (above/below), and value.

Actions

1
Send Test — sends a test message to the digest webhook immediately. Useful to verify the webhook URL is correct.
2
Send Digest Now — triggers a full digest delivery outside of the scheduled time.
3
Check Thresholds Now — manually runs the threshold check and fires any matching alerts.

☁️ Azure AD 🔒 Admin Only

Configure OIDC / OAuth2 single sign-on via Microsoft Azure Active Directory. Only available to users with an admin role.

FieldDescription
Tenant IDAzure AD Tenant ID (GUID) from the App Registration Overview page.
Client IDApplication (client) ID from the App Registration.
Client SecretSecret generated in Certificates & Secrets. Stored server-side only.
Redirect URLOAuth2 callback URL. Must be registered in Azure AD as an allowed redirect URI. Format: https://yourserver/auth/callback.
🔴 Server restart required after saving Azure AD settings. The OIDC client is initialized at startup and must be restarted to pick up the new credentials.

Azure AD App Registration Steps

1
Azure Portal → App registrations → New registration — name the app "AV Dashboard".
2
Authentication → Add platform → Web — enter the Redirect URL (must match exactly).
3
Certificates & secrets → New client secret — copy the value immediately (shown once).
4
Token configuration → Add groups claim — select "Security groups" so group memberships are available for role mapping.
5
Paste Tenant ID, Client ID, Client Secret, and Redirect URL into Settings → Azure AD, then Save and restart the server.
🗝 Role Mappings 🔒 Admin Only

Maps Azure AD identities (groups, emails, or UPN patterns) to dashboard roles. Evaluated in order — first match wins.

ConceptDescription
Match TypeGroup (Azure AD Group Object ID), Email (exact match), or Pattern (regex/wildcard on UPN/email).
Match ValueAzure AD Group GUID, email address, or pattern string.
RoleDashboard role to assign: exec, rte, pm, sm, admin.
Admin RolesComma-separated list of role names that get access to admin-only Settings tabs.
ℹ️ Users with no matching role mapping are assigned the all role (read-only, all sections visible). Add a catch-all mapping at the bottom if you want authenticated users to default to a specific role.
🛡 Visibility Policies 🔒 Admin Only

Fine-grained control over what each role can see. Policies hide pages, tabs, and individual chart cards without removing them from the codebase.

How It Works

1
Select a role from the role picker (All, Exec, RTE, PM, SM, Admin).
2
Each section card shows its current visibility status. Click ✓ Visible / ⊘ Hidden to toggle the entire section for that role.
3
Expand ▼ a section card to see individual tabs (📑) and charts (📊) as toggleable chips.
4
Save Policies — changes take effect on the next page load for users of that role.
