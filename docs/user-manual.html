🚀 1. Getting Started

AV Dashboard is a React + Node.js web application that connects to Team Foundation Server (TFS / Azure DevOps) and presents live PI delivery data for the your engineering programme. It is designed for daily operational use: monitoring feature flow, delivery predictability, quality, risks, objectives, and release readiness from one dashboard.

ItemRecommendationWhy it matters
BrowserGoogle Chrome or Microsoft Edge (latest stable)Best compatibility for large charts, sticky navigation, modal dialogs, and fullscreen cards.
Server runtimeNode.js 18+The local dashboard server runs on Node.js and serves both API responses and the React application.
Application URLhttp://localhost:3000This is the default address for the locally hosted dashboard.
Data sourceTFS / Azure DevOpsAll delivery metrics are derived from live work item data, saved queries, classification nodes, and snapshot files.

Open the dashboard

1
Open Chrome or Edge

Use a modern Chromium-based browser to ensure all dashboard controls render correctly.

2
Navigate to the local dashboard URL
url
http://localhost:3000

If the server is running, the dashboard shell loads immediately and begins requesting live programme data.

3
Wait for the first refresh cycle

The initial load can take a few moments while the server reads configuration, connects to TFS, and resolves the available PI list, teams, and dashboard payloads.

Complete first-time setup

1
Open the Settings page

Select Settings from the top bar or the dashboard navigation.

2
Enter your TFS base information

Provide the TFS base URL, a valid Personal Access Token (PAT), and the programme area path used to scope work items.

3
Save the configuration

After saving, use the refresh action to reload live data with the new connection settings.

4
Verify the top bar filters

Confirm that PI options, team filters, and role views appear. If they do, the dashboard is successfully reading configuration and base data.

🔐 PAT Security

Your PAT is sensitive. Store it only in the dashboard settings/configuration, never in screenshots, emails, chat messages, or shared documents.

Understand the dashboard shell

Dark theme

The application uses a dark dark theme optimized for wallboards, meetings, and day-to-day monitoring. Strong color contrast makes RAG statuses and chart trends easy to scan quickly.

Navigation bar

The top navigation bar is the main control surface for PI selection, team filtering, role view switching, snapshot selection, refresh, and settings access.

📋 3. Executive Section

The Executive section is the programme summary page. It is designed for fast health assessment and is typically the first stop for weekly operational reviews, leadership updates, and PI health discussions.

CardWhat it showsHow to interpret it
Programme Health Score0–100 arc gauge with green, amber, and red zones.Green is greater than 70, amber is 40–70, and red is below 40. Treat it as a composite signal for delivery health, not a standalone verdict.
PI Programme ScoreSprint progress bars for S1, S2, S3, and IP with effort points and feature counts.Use it to see whether delivery is progressing evenly through the PI or slipping toward later sprints/IP.
Committed vs DeliveredCompares features committed during planning versus features actually delivered.If delivered is materially below committed, investigate predictability, scope movement, blocked dependencies, or quality issues.
Team ScorecardPer-team health metrics in one summary card.Use it to identify outliers quickly and decide which teams require deeper review.

How to read the score

  • Green (>70): Delivery is broadly healthy and within expected tolerance.
  • Amber (40–70): Review scope, WIP, defects, or PI checks because the programme is under some pressure.
  • Red (<40): Immediate attention is required. Drill into Features, Defects, Risks, and Dependencies to isolate the cause.
⚠ Interpretation guidance

A low score should always be cross-checked against the underlying cards. For example, a weak committed-vs-delivered signal plus rising defect aging usually points to execution strain rather than a data issue.

🚀 4. Features Section

The Features section is the core delivery workspace. It combines high-level KPIs, flow analytics, saved PI checks, predictability against a planning snapshot, and cross-team dependency visibility.

TabMain purposeBest used for
OverviewLive feature delivery and flow metrics.Weekly delivery reviews, team follow-up, and programme execution monitoring.
PI ChecksSaved query compliance checks against PI readiness and hygiene.Governance, planning quality, and backlog hygiene reviews.
PredictabilityCompares live delivery against a PI Plan Data snapshot.Assessing committed vs stretch completion and team predictability.
DependenciesShows cross-team feature dependencies.Finding blocked delivery chains and unresolved hand-offs.

Overview tab

WidgetDescriptionTypical action
Cycle Time Distribution (last 4 PIs)Histogram of feature cycle times over the last four PIs with a Gaussian overlay and KPI strip (avg / median / p25 / p75 / σ). Always shows the last 4 PIs — independent of the PI selector.Use to understand delivery pace across recent programme history. A narrow distribution centred below 30 days indicates a healthy flow pattern.
Total Features / Features Done / Done Rate %Core outcome KPIs for the selected PI scope.Use these first to establish whether the PI is trending toward completion.
WIP count / Stale FeaturesShows active work volume and items not moving.High WIP or stale counts often indicate overcommitment, dependency wait states, or ownership gaps.
Feature Lifecycle FunnelHorizontal bar chart of all states: Forecasted → New → Approved → Activated → Done → Removed.Click any bar to open the matching TFS query and inspect underlying work items.
Team SummaryPer-team delivery chart with Count / Points (man-days) toggle (defaults to Points). In Points view: stacked bars show Done + In Progress + Remaining effort (total = Planned), paired with a Dev Capacity bar from TFS — both in man-days (1 day = 6 hours by default, configurable in Settings).Use Count for workload volume. Use Points to compare how much effort each team has planned against their available development capacity for the PI. A team whose Planned stack exceeds Capacity is over-committed.
Cycle Time SummaryAverage, minimum, and maximum days from Created → Done.Use this to judge delivery efficiency and identify unusually long-lived items.
Feature Age DistributionAge of non-Done features in their current state.Useful for spotting items stuck in Forecasted, Approved, or Activated for too long.
Cross-Team DependenciesDelivery links spanning multiple teams.Use it to identify blockers before they turn into slipped features.
ℹ Man-days unit in Team Summary

The Effort field on a Feature work item in TFS represents man-days. Dev Capacity hours are divided by the Hours per Man-Day setting (default: 6) to produce the same unit. This means the Planned stack and the Capacity bar are directly comparable — both measure how many days of work are in scope versus available.

ℹ Click-through behaviour

Most Overview charts are interactive. Clicking a bar, segment, or ID takes you directly to a filtered TFS result so you can move from summary to action without rebuilding the query manually.

PI Checks tab

The PI Checks tab runs 17 checks based on TFS saved queries. Each check returns a Red/Amber/Green status and the number of matching items.

StatusMeaningUser action
GreenNo significant issue detected for the saved query.Monitor only; no immediate intervention needed.
AmberSome issues exist and should be reviewed.Open the check details, inspect the item list, and assign clean-up actions.
RedHigh-priority data or planning hygiene issue.Open the TFS link immediately and address the affected items before the next planning checkpoint.

  2. Review the RAG state and issue count for each check.
  3. Click the check name to view the detailed item list.

Read the comparison table

If the Predictability tab looks empty, first confirm that a snapshot exists for the PI and that it is selected in the 📋 PI Plan Data control.

👥 6. Teams Section

The Teams section organizes feature and defect information by team so you can quickly move from programme-level concern to team-level ownership.

ElementDescriptionWhy it matters
Per-team feature chartsShows delivery state and throughput for each team.Helps identify which teams are moving work to Done and which are carrying high WIP.
Per-team defect chartsDisplays quality load by team.Useful for balancing delivery pressure against defect pressure.

How team area paths work

  1. The dashboard discovers teams from configured TFS area path roots.
  2. Each team filter option maps back to an Area Path or branch of Area Paths in TFS.
  3. When a team is selected, compatible widgets rebuild queries using that team path.
  4. If a team is missing or the filter behaves unexpectedly, review the teamRootPath configuration in Settings.

🗺 11. Roadmap Section

The Roadmap section visualizes feature delivery by team and sprint over time. It is ideal for release sequencing, PI planning follow-up, and cross-team scheduling conversations.

Roadmap elementHow to read it
RowsTypically represent teams, feature groups, or roadmap tracks.
ColumnsRepresent time buckets such as sprints or PI periods.
Colored itemsShow where a feature is planned, active, or expected to land.
Gaps or crowdingMay indicate idle periods, overload, or unrealistic sequencing.

How to read the roadmap

  1. Start with the left-hand row label to identify the team or workstream.
  2. Move across the timeline from early sprint columns to later sprint columns.
  3. Check whether critical features are clustered too late in the PI or overlap with many other items from the same team.
  4. Use the roadmap together with Dependencies and Risks to confirm the sequencing is achievable, not just visually planned.

🔗 12. Dependencies Section

Dependencies expose feature relationships that cross team boundaries. This view is valuable during programme syncs because blocked dependencies are often the hidden cause of slips and low predictability.

StatusMeaningRecommended action
BlockedThe dependent feature cannot complete as planned because another team or item has not delivered the prerequisite.Escalate during PI sync, assign ownership, and review target sprint alignment.
ResolvedThe dependency condition has been satisfied.Confirm downstream work has resumed and remove the blocker from action lists.

Review dependencies effectively

  1. Sort or scan the list for blocked items first.
  2. Check which team owns the prerequisite and which team is waiting.
  3. Validate that both teams agree on the target sprint and status.
  4. Use the linked TFS items to confirm whether the blocker is real, outdated, or already resolved.

🏥 14. Release Health Section

Release Health summarizes readiness signals for an upcoming or in-progress release. Treat it as the answer to the question, “If we released now, how prepared are we?”

Typical metricWhat it indicates
Readiness scoreOverall confidence based on delivery, quality, test, and risk signals.
Open blockersCritical unresolved conditions that prevent safe release.
Quality trendWhether defects and test outcomes are moving toward acceptable release condition.

Recommended review flow

  1. Start with the readiness score to understand overall release posture.
  2. Check blockers next; a single blocker can outweigh otherwise healthy trend signals.
  3. Use defect, risk, and test coverage sections to validate whether the readiness score matches operational reality.

📊 16. Scope Change Tracking

The Scope Change page compares a previously captured snapshot (your PI planning baseline) against the current live PI scope in TFS. It answers the key question: "How much has our planned scope changed since PI planning?"

ComponentWhat it showsWhen to use it
Snapshot selectorLists all available snapshots. Each row shows PI labels, capture timestamp, and feature count.Select the PI planning snapshot as your baseline, then click Compare vs Now.
KPI stripBaseline points, current points, net growth %, scope churn %, added points, removed points, and RAG risk status.Glance at whether scope has grown or shrunk and whether that growth is within acceptable bounds.
Scope Points barSide-by-side bars for Baseline, Current, Added, and Removed story points.Visualize the magnitude of scope movement relative to the original plan.
Change Breakdown panelCounts of each change type: Added/Moved-In, Removed, Estimate Changed, Sprint Moved, Team Changed.Quickly identify which category of change is driving scope instability.
Change Detail tableEvery changed feature with type, ID (TFS link), title, change badge, detail (old → new), points, and owner. Tabbed by change type.Drill into specific items, open TFS links for investigation, and track ownership of scope changes.

How risk status is calculated

StatusConditionWhat it means
GreenNet growth ≤ 5% and churn ≤ 25%Scope is stable and within normal variance from PI planning.
AmberNet growth 5–15% or churn 25–40%Meaningful scope movement. Review with the RTE and product management.
RedNet growth > 15% or churn > 40%Significant scope change. Stakeholder communication and PI scope review required.

Change types explained

BadgeMeaning
AddedFeature was created after the baseline snapshot date — new scope not in the original PI plan.
Moved InFeature existed before the snapshot but was pulled into this PI after the baseline was taken.
RemovedFeature was in the baseline but is no longer present in the current PI scope (state changed to Removed, or pulled out).
Estimate UP / DOWNThe effort (story points) on a feature changed relative to the baseline value.
Sprint MovedThe iteration path of a feature changed after the baseline — sprint re-assignment.
Team ChangedThe area path / owning team of a feature changed after the baseline.

Step-by-step: how to run a scope comparison

1
Navigate to Scope Change

Select 📊 Scope Change from the left navigation sidebar.

2
Choose a baseline snapshot

The snapshot list shows all previously captured PI snapshots. Select the one taken during or at the end of PI planning for the PI you want to evaluate.

3
Click "Compare vs Now"

The server fetches the current PI scope from TFS, compares it item-by-item against the snapshot, and returns the full change set. This may take a few seconds on first run — a sidecar cache is saved for faster subsequent loads.

4
Review the KPI strip

Check net growth % and scope churn % first. The RAG badge gives an immediate health signal.

5
Drill into specific changes using tabs

Use the Added, Removed, Estimate Change, Sprint Move, and Team Change tabs to filter the change table by type. Click TFS IDs to open the work item directly in TFS.

ℹ Snapshots as baselines

The Scope Change page reuses the same snapshots you already capture in PI Delivery. No separate baseline capture is needed. Take a snapshot at the end of PI planning; that becomes your scope baseline for the entire PI.

⚠ First compare is slower

On first comparison, the server fetches and caches the raw feature list from the snapshot's PI. Subsequent comparisons use the cached sidecar file and are much faster.

📋 18. Snapshots

A snapshot is a point-in-time capture of the PI plan. It freezes planning state so the dashboard can later compare live delivery against the original baseline.

Take a snapshot

1
Click 📋 PI Plan Data in the top bar

This opens the snapshot selector and management panel.

2
Select the PI you want to capture

Choose the PI that represents the planning baseline you want to preserve.

3
Click Capture

What a snapshot captures

Captured dataUsed for
FeaturesPredictability, committed-vs-delivered analysis, and PI baseline comparison.

ObjectivesObjective linkage and planning comparison.

Use snapshots for predictability and test coverage delta

  1. Select a snapshot in the top bar.
  2. Go to Features → Predictability to compare planned versus live feature completion.
  3. Go to the Test Coverage view if you want to review coverage delta since the snapshot was taken.

Manage snapshots

  • Open the PI Plan Data panel to see the snapshot list.
  • Select a snapshot to make it active for comparison views.
  • Delete outdated or incorrect snapshots when they are no longer required.
  • Snapshots are stored in the project’s snapshots/ folder.
⚠ Baseline discipline

Capture the first snapshot immediately after PI planning. If you capture only late revisions, predictability views will no longer represent the original commitment baseline.

🔗 19. TFS Links

Deep linking is a core AV Dashboard feature. Most charts are not just visual summaries; they are launch points into the exact TFS work item subset behind the number.

Link sourceResult
Chart bar / slice / segmentOpens a filtered TFS query in a new browser tab.
Table ID link (#12345)Opens the specific TFS work item.
Saved PI check linkOpens the source saved query or related work item list in TFS.

How query filters are built

wiql
WHERE AreaPath      UNDER configured programme/team path
  AND IterationPath UNDER selected PI or sprint path
  AND State         IN selected chart state(s)
  AND WorkItemType  = Feature or Defect

In practice, the dashboard injects the currently selected area path, PI, sprint, and state values so users land directly on the relevant records instead of a generic backlog view.

🔧 21. Troubleshooting

If a section looks empty or inconsistent, start by checking filters, snapshot selection, and TFS configuration. Most issues are caused by expired credentials or incorrect path settings.

ProblemLikely causeResolution
PAT expiredThe saved TFS token is no longer valid.Generate a new PAT in TFS, update Settings, save, and refresh the dashboard.
No dataIncorrect area path or iteration path.Verify programme scope in Settings and ensure the configured paths match TFS exactly.
Charts empty after refreshData load still in progress or server refresh still running.Wait briefly and refresh again if needed. For large queries, allow extra time for TFS responses.
Team filter not workingteamRootPath is wrong or incomplete.Review the team root configuration and confirm the expected teams exist under that path in TFS.
Snapshot not foundThe expected snapshot file does not exist.Check that the snapshots/ folder exists and that the required PI snapshot was captured successfully.
⚠ Fast triage checklist

When in doubt, confirm these four items in order: selected PI, selected team, active snapshot, and TFS settings. Most apparent dashboard issues come from one of those filters or inputs.
